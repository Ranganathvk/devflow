# /convention-detect — Coding conventions (Phase 0)

## Purpose

Capture **how this repo already writes code** so feature work matches existing patterns — not alien abstractions.

This skill samples representative files guided by `PROJECT_OVERVIEW` boundaries and records conventions with **evidence paths**. It does not prescribe new standards or refactor code.

## When to invoke

- `/convention-detect` — immediately after `/workspace-scan` (or when overview exists but conventions are missing/stale).
- When `artifacts/CONVENTIONS.*` is missing or **stale** after large codebase style changes.
- Do **not** invoke before `PROJECT_OVERVIEW` exists — run `/workspace-scan` first.
- Do **not** use for repo layout discovery — that is `/workspace-scan`.

## Inputs

- **Required:** `artifacts/PROJECT_OVERVIEW.contract.yaml` — read in full for `modules`, `languages`, `frameworks`, and `test` hints.
- **Required (fallback):** `artifacts/PROJECT_OVERVIEW.md` — only if contract is missing; then stop after writing conventions and note contract should be regenerated via `/workspace-scan`.
- **Required:** Representative source samples from the consumer repo (see workflow).
- **Optional:** Linter/formatter configs (`.editorconfig`, `eslint*`, `prettier*`, `ruff.toml`, `checkstyle.xml`, etc.).
- **Optional:** `artifacts/SPEC.md` — cross-check only; never override code evidence.
- **Forbidden:** Unbounded chat history; reading every file in the repo; inventing conventions without evidence; editing application source.

## Workflow

1. Load `artifacts/PROJECT_OVERVIEW.contract.yaml`. If missing, stop and direct human to `/workspace-scan`.
2. Build a **sampling plan** from `modules[]` (max **4 modules**, prioritize those marked as application/core over tooling if many exist):
   - Per module: pick **2–4 files** total across the convention areas below (reuse files when one file shows multiple patterns).
   - Prefer: one service/handler, one model/DTO/entity (if layer exists), one test file, one config entrypoint.
3. For each convention area, inspect samples and record **observed patterns** (not ideals):

   | Area | Look for |
   |------|----------|
   | **Naming** | Files, types, functions, constants, test names; package/module naming |
   | **Dependency injection** | Constructor injection, frameworks (`@Injectable`, `wire`, Spring `@Autowired`, manual factories), service locators |
   | **DTO / entity style** | Request/response types, ORM models, immutability, validation annotations, mapping layers |
   | **Exception handling** | Custom exception types, HTTP/error mapping, `Result` types, try/catch patterns at boundaries |
   | **Logging** | Logger acquisition, levels, structured fields, correlation IDs |
   | **Config** | Env vars, config files, secrets handling, profiles, feature flags |
   | **Testing** | Framework, file layout (`*_test`, `*.spec`), fixtures, mocks, assertion style, coverage hooks |

4. Use IDE search (symbols, references) sparingly to confirm a pattern appears in **more than one** file when claiming a repo-wide rule; otherwise label **local / module-only**.
5. Write `artifacts/CONVENTIONS.md` using the skeleton below (target **≤ ~150 lines**).
6. Write `artifacts/CONVENTIONS.contract.yaml` using the YAML shape below.
7. Cross-check: each `conventions.*` entry has `evidence[]` or is listed in `open_gaps`.
8. **STOP.** Chat reply: paths written, bullet summary per convention area, numbered `open_gaps`, suggested next step: `/understand "<change>"` or `/design <FEATURE>` per [BROWNFIELD_DEV_LOOP.md](../../docs/BROWNFIELD_DEV_LOOP.md). Do **not** chain feature design or implementation.

## Output artifacts

| Path | Change | Notes |
|------|--------|-------|
| `artifacts/CONVENTIONS.md` | Created or replaced | Human-readable conventions; ≤ ~150 lines target |
| `artifacts/CONVENTIONS.contract.yaml` | Created or replaced | Machine handoff for feature skills |

No other files written or edited.

## `CONVENTIONS.md` skeleton

```markdown
# Coding conventions

> Convention map. Evidence-backed; generated by /convention-detect.

## Document control

| Field | Value |
|-------|-------|
| Detected | YYYY-MM-DD |
| Overview ref | artifacts/PROJECT_OVERVIEW.contract.yaml |
| Revision | initial \| refreshed |

## 1. Summary

One paragraph: dominant style and any notable splits (e.g. core vs adapters differ).

## 2. Naming

| Scope | Pattern | Example | Evidence |
|-------|---------|---------|----------|
| Files | … | … | path |
| Types / classes | … | … | path |
| Functions / methods | … | … | path |
| Constants | … | … | path |
| Tests | … | … | path |

## 3. Dependency injection

How dependencies are provided and consumed. Note framework vs manual.

## 4. DTO / entity style

How data crosses layers; validation; mapping.

## 5. Exception handling

Error types, propagation, API/boundary mapping.

## 6. Logging

Logger setup, levels, structured logging habits.

## 7. Configuration

Config sources, env naming, secrets, profiles.

## 8. Testing

Layout, frameworks, mocks, naming, CI hooks (from overview if relevant).

## 9. Tooling enforcement

Linter/formatter rules that reinforce conventions (or "none found").

## 10. Module-specific notes

Only if patterns differ by boundary; cite paths.

## 11. Open gaps

What needs deeper inspection before implementing in that area.
```

## `CONVENTIONS.contract.yaml` shape

```yaml
contract_version: "1"
artifact: conventions
workflow_profile: devflow
detected_at: "YYYY-MM-DD"
project_overview_contract_path: artifacts/PROJECT_OVERVIEW.contract.yaml

summary: "<one paragraph>"

conventions:
  naming:
    scope: repo | module | mixed
    patterns: []
      # - id: file_snake_case
      #   statement: "Python modules use snake_case filenames"
      #   example: "user_service.py"
      #   evidence: ["src/services/user_service.py"]
    notes: null

  dependency_injection:
    scope: repo | module | mixed | none_found
    patterns: []
    notes: null

  dto_entity:
    scope: repo | module | mixed | none_found
    patterns: []
    notes: null

  exceptions:
    scope: repo | module | mixed | none_found
    patterns: []
    notes: null

  logging:
    scope: repo | module | mixed | none_found
    patterns: []
    notes: null

  config:
    scope: repo | module | mixed | none_found
    patterns: []
    notes: null

  testing:
    scope: repo | module | mixed | none_found
    patterns: []
    notes: null

tooling:
  linter: null
  formatter: null
  evidence: []

module_overrides: []
  # - module_id: adapters
  #   area: naming
  #   statement: "..."
  #   evidence: []

open_gaps: []

downstream_skills: []
  # e.g. design
```

**Pattern object fields:** `id`, `statement`, `example` (optional), `evidence` (non-empty for confirmed patterns).

**Parsing note:** Use `none_found` scope when the layer does not exist in this repo (e.g. no ORM). Use `open_gaps` for uncertainty, not silent guesses.

## Context budget

- Read in full: `PROJECT_OVERVIEW.contract.yaml`, linter config headers (not every rule file line-by-line).
- Read partially: **≤ 12 source/test files** total across the run unless human expands scope.
- Route sampling through `modules[]` from overview — do not wander unrelated trees.
- Do **not** load full prior chat or entire directories.

## Failure handling

- **Missing `PROJECT_OVERVIEW.contract.yaml`** — Stop; require `/workspace-scan` first.
- **Overview lists no modules** — Sample from repo root + primary language default paths (`src/`, `lib/`, `app/`, `internal/`); record wide sampling in `open_gaps`.
- **Polyglot repo with conflicting styles** — Document per `module_overrides`; set area `scope: mixed`.
- **Insufficient code** (docs-only repo) — Emit minimal conventions with all areas in `open_gaps`.

## Forbidden

- Editing source, tests, or build configs.
- Rewriting `PROJECT_OVERVIEW.*`.
- Feature spec, impact, design, or implementation artifacts.
- Prescribing new conventions the repo does not use.
- Chaining `/design` or `/implement` in the same invocation.

## Quality bar (self-check before finishing)

- [ ] Both artifact paths exist; YAML parses.
- [ ] Every `patterns[]` entry includes at least one `evidence` path.
- [ ] Conflicts between modules are explicit (`module_overrides` or `scope: mixed`).
- [ ] Chat ends with STOP + how to start feature work (feature ID + next skill when available).
