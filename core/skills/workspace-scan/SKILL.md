# /workspace-scan — Workspace orientation (brownfield Phase 0)

## Purpose

Produce a **concise, evidence-backed** snapshot of an **existing** codebase so later brownfield skills can work surgically without whole-repo dumps.

This skill discovers stack, build/test tooling, module boundaries, and package layout from **repo facts** — not from chat memory or speculative architecture redesign.

## When to invoke

- `/workspace-scan` — first step on an attached brownfield repo (before `/convention-detect`).
- When `AI_CONTEXT/PROJECT_OVERVIEW.*` is missing or **stale** after major repo restructuring.
- Do **not** invoke when only refreshing coding conventions — use `/convention-detect` instead.
- Do **not** use for greenfield product shaping — use `/grillme` and `/system-hld` instead.

## Inputs

- **Required:** Consumer repository tree (root and immediate children; build/config manifests).
- **Optional:** `AI_CONTEXT/SPEC.md` (if present — use only to cross-check product name/boundary, not as primary evidence).
- **Optional:** Existing `AI_CONTEXT/PROJECT_OVERVIEW.md` (refresh mode only when human asks to preserve intentional edits).
- **Forbidden:** Unbounded chat history; loading entire source trees; inventing modules not evidenced by paths or config; redesigning architecture.

## Workflow

1. Ensure `AI_CONTEXT/` exists; create the directory if missing.
2. **Orientation pass (surgical):** Inspect only what bounds the repo:
   - Root: `README*`, `package.json`, `pom.xml`, `build.gradle*`, `Cargo.toml`, `go.mod`, `pyproject.toml`, `requirements*.txt`, `Makefile`, `docker-compose*`, `*.sln`, `Directory.Build.props`, etc.
   - Top-level directories (one level); note obvious app vs library vs infra splits.
   - CI/config hints: `.github/workflows`, `Jenkinsfile`, `.gitlab-ci.yml`, `azure-pipelines.yml`.
   - Test entrypoints: `jest.config*`, `vitest.config*`, `pytest.ini`, `*_test.go` patterns in test dirs (count dirs, do not read all tests).
3. For each **major boundary** identified, read **at most** 1–2 representative files (e.g. main entry, root module `__init__`, or a single package README) to confirm role — not every file in the tree.
4. Record findings with **evidence paths** (file or directory). Unknowns → `TBD` or `open_gaps` — never guess frameworks from folder names alone without a manifest.
5. Write `AI_CONTEXT/PROJECT_OVERVIEW.md` using the skeleton below (target **≤ ~120 lines**).
6. Write `AI_CONTEXT/PROJECT_OVERVIEW.contract.yaml` using the YAML shape below.
7. Cross-check: every contract list item traces to a path in `evidence` or is explicitly in `open_gaps`.
8. **STOP.** Chat reply only: paths written, top-level boundaries (bullets), numbered `open_gaps`, suggested next: `/understand` (if conventions also needed) or `/convention-detect` (conventions only). Do **not** chain feature work in the same invocation.

## Output artifacts

| Path | Change | Notes |
|------|--------|-------|
| `AI_CONTEXT/PROJECT_OVERVIEW.md` | Created or replaced | Concise orientation; ≤ ~120 lines target |
| `AI_CONTEXT/PROJECT_OVERVIEW.contract.yaml` | Created or replaced | Machine handoff for downstream brownfield skills |

## `PROJECT_OVERVIEW.md` skeleton

```markdown
# Project overview

> Brownfield orientation. Evidence-backed; generated by /workspace-scan.

## Document control

| Field | Value |
|-------|-------|
| Scan date | YYYY-MM-DD |
| Repo root | `<path or .>` |
| Revision | initial \| refreshed |

## 1. Summary

One short paragraph: what this repo appears to be (app, library, monorepo, framework, etc.).

## 2. Stack & runtime

| Area | Finding | Evidence |
|------|---------|----------|
| Primary language(s) | … | path |
| Framework(s) | … or TBD | path |
| Runtime / SDK | … or TBD | path |

## 3. Build & tooling

| Tool | Role | Evidence |
|------|------|----------|
| Build | … | path |
| Package manager | … | path |
| Lint / format | … or none found | path |

## 4. Test & quality

| Tool | Scope | Evidence |
|------|-------|----------|
| Test framework | … | path |
| CI | … or none found | path |

## 5. Module structure & boundaries

Table: id, path, role, notes

## 6. Package / namespace layout

Short bullets or table for how code is organized (packages, crates, modules).

## 7. Major integrations (if evident)

External systems, databases, queues — only if config or README states them; else "none found".

## 8. Architecture style (tentative)

One paragraph labeled **guess from layout** — must cite evidence paths; no redesign proposals.

## 9. Open gaps

Bullets: what could not be determined without deeper inspection.
```

## `PROJECT_OVERVIEW.contract.yaml` shape

```yaml
contract_version: "1"
artifact: project_overview
workflow_profile: brownfield
scanned_at: "YYYY-MM-DD"
repo_root: "."

summary: "<one paragraph, evidence-backed>"

languages: []
  # - id: typescript
  #   evidence: ["package.json"]

frameworks: []
  # - id: express
  #   evidence: ["package.json"]
  #   confidence: confirmed | tentative

build:
  tool: null
  package_manager: null
  evidence: []

test:
  framework: null
  ci: null
  evidence: []

modules: []
  # - id: api
  #   path: "src/api"
  #   role: "HTTP API"
  #   evidence: ["src/api/main.ts"]

boundaries: []
  # Short strings naming major separation lines (e.g. "core vs adapters")

integrations: []
  # Only if evidenced; else []

architecture_guess: null
  # One line + evidence paths, or null

open_gaps: []
  # Strings — unknowns, not guesses

downstream_skills:
  - convention-detect
```

**Parsing note:** Use YAML `null` for unknown scalars; empty arrays when nothing found. Keep the file **small** (typically under 80 lines).

## Context budget

- Read **fully:** root README (if present), primary build manifest(s), CI workflow summary (first workflow file only unless monorepo needs one per package).
- Read **partially:** 1–2 representative files per major boundary.
- Do **not** read: entire `node_modules`, `vendor`, generated trees, or all source files.
- Prefer workspace search / symbol tools over opening large directories.

## Failure handling

- **Empty or non-code repo** — Write minimal overview with `open_gaps` explaining insufficient evidence; still emit both artifacts.
- **Monorepo with multiple manifests** — List each top-level package as a `modules` row; do not collapse into one false stack.
- **Conflicting evidence** (e.g. two lockfiles) — Record conflict in `open_gaps`; pick `TBD` for disputed fields.

## Forbidden

- Editing application source, tests, or build configs.
- Creating `CONVENTIONS.md` (that is `/convention-detect`).
- Feature design, implementation, or `FEATURE_SLICES` artifacts.
- Mega-output prose or dumping directory trees into markdown.
- Continuing to `/convention-detect` in the same run — human gate between Phase 0 steps.

## Quality bar (self-check before finishing)

- [ ] Both artifact paths exist; YAML parses.
- [ ] Every non-trivial claim has an `evidence` path in prose table or contract.
- [ ] `architecture_guess` is labeled tentative and cites layout evidence.
- [ ] Chat ends with STOP + suggested `/convention-detect` only.
